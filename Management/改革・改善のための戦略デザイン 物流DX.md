# 改革・改善のための戦略デザイン 物流DX

## 0章 DXの本質

### 0-1 フォロワー・ニッチャーが取り組むプロセスイノベーションDX

- DXブーム到来の背景
  - 2018年：経済産業省「デジタルトランスフォーメーション（DX）を推進するためのガイドライン」
  - 米国巨大IT企業の躍進
- DXの種類と本書の対象
  1. ビジネスイノベーション
     - デジタル技術を活用して、新たな市場・顧客を創る
     - 新たな需要を創出し、事業の売上・利益拡大に寄与する
  2. プロセスイノベーション
     - デジタル技術を活用して、自社・業界の業務プロセスを変革する
     - 業務効率性を高めるBPR（Business Process Engineering：ビジネスプロセスの再構築）を実現し、主に事業の運営コストを削減、利益の改善に寄与する
     - フォロワー・ニッチャー企業向け＝本書の領域

### 0-2 流行り言葉に要注意――本質思考の重要性

- ERP全盛期とDXの類似性
  - 業務再構築（BPR）→ パッケージ型機関情報システムERP（Enterprise Resource Planning：基幹業務（販売、生産、物流、会計など）のITシステムの総合パッケージソフトウェア）の導入にすり替わる事例が多発
  - 業務再構築の目的を堅持 → 成功
  - 部分的なシステムの置き換え → 投資効果が不明確
  - 本質志向が重要
- 逆タイムマシン経営論に学ぶ志向の罠：流行りの情報や言説に流されて意思決定を誤る
  - 3つの志向の罠
      1. 飛び道具のトラップ：銀の弾丸は存在しない
      2. 激動期のトラップ：乗り遅れる錯覚に陥る
      3. 遠近歪曲のトラップ：欧米で主流の手法を当社でも見習わなけらばという錯覚に陥る
- 自社の文脈に合ったDXの取り組みを
  - DXを単純に放棄しない
  - 自社の戦略とDXの目的を見据えて、地に足のついた独自のDXを実践する

### 0-3 DXによる物流改革

- DXの目的は経営・業務改革
  - DX：競争優位を実現する経営・業務における変革をデジタルという手段を使って実現する
- DXとITの違い
  - IT
    - アナログで行っていた業務のデジタル化 → 作業効率のアップ
    - 手段志向
  - DX
    - ビジネスモデル・オペレーションの変革が発想の起点
    - 目的志向

## 1章 物流が企業戦略の生命線に

### 1-1 SCMと物流について

- SCMとは「需要と供給の管理」
  - 「顧客に、必要なモノを、必要な時に、必要な場所に、必要な量だけ届ける厳密な供給の管理ができる仕組み作り・運用」
- SCMのメリット
  1. 適切な需要変動への対応：機会損失の防止・顧客満足度アップ
  2. 在庫の最適化：在庫過多／過小のリスク防止・機会損失の防止
  3. 納期リードタイムの短縮：顧客満足度の向上・機会損失の防止
  4. 物流ソフトの削減：在庫管理コスト削減・輸配送コスト削減
- SCMが機能する鍵は物流にある
  - 物流：欠かすことのできない構成要素

### 1-2 SCMが再認識されてきた

- 経営課題であるSCMが、システム導入にすり替わってしまった
  - IT業界：SCM＝高度に自動化された需給計画・管理システムの導入があ必要
  - SCM＝情報システムの課題
- グローバルでの大きな環境変化でSCMが再浮上
  - コロナ・パンデミック、ロシア・ウクライナ紛争、米中貿易摩擦、洪水など
  - 需給計画・管理に関する活動デザイン、利害関係者との調整など事業活動のマネジメント＝SCMの本質と再認識
- SCMが再注目された要因
  1. 急激な需要変動の発生＝在庫の最小化が善から変化
  2. 調達・生産の制約条件の変化＝供給先との関係強化
  3. 物流制約条件の変化＝**物流力の強化**が企業の競争力に

### 1-3 物流がサプライチェーンのボトルネックになることが認識されてきた

- サプライチェーンの性能は制約条件（ボトルネック）に同期する
- サプライチェーン性能アップは、ボトルネックの改善！
- サプライチェーンのボトルネック＝**物流**
  - あるべき姿のサプライチェーン（ToBeモデル）→ モノの保管場所や移動経路の見直しが必須
  - **物流**＝サプライチェーンに検討における最重要テーマの一つ
- 物流がSCM全体の制約となる＝物流が企業戦略の生命線になる

### 1-4 サプライチェーンの拡大・進化に伴う物流の変遷

### 1-7 SCMトピック：サプライチェーン強靭化への取り組み――自動車業界(1)

    3. 確定注文期間や、サプライヤー側で在庫が積みあがってしまった場合の買取保障のあり方を見直す
    4. 仕様変更時の評価プロセスを完成車メーカー間で標準化する

  5. 上海ロックダウン強化による部品調達への影響発生（22年4～6月）

## 3章 物流DXの定義

### 3-1 DXの定義

- ウメオ大学 ストルターマン教授（2004年）
  >「ITの浸透が、人々の生活をあらゆる面でより良い方向に変化させる」
- 経済産業省「デジタルトランスフォーメーション（DX）を推進するためのガイドライン」（2018年）
  >「企業がビジネス環境の激しい変化に対応し、データとデジタル技術を活用して、顧客や社会のニーズを基に、製品やサービス、ビジネスモデルを変革するとともに、業務そのもや、組織、プロセス、企業文化・風土を改革し、競争上の優位性を確立すること」
- DXの目的
  - 経営・業務課題の解決が目的
  - デジタルの活用はあくまでも手段

### 3-2 物流DXに取り組む上での心構え

- 国土交通省
  >「サプライチェーン全体での機械化・デジタル化により、情報・コスト等を『見える化』、作業プロセスの単純化・常態化し、物流のこれまでの在り方を変革する」
- フォロワー・ニッチャーでも物流DXに取り組める
  - 新しい技術導入ではなく、経営・業務課題を解決すること
  - 自社にあった本質志向のDXに取り組む

### 3-3 オペレーション・事業インフラが事業競争力の違いを生む

- ビジネスモデルだけでは企業の収益性の差異が説明できない
  - 例：サブスクリプションモデル＝昔からある
- 収益性の差異
  - オペレーションと事業インフラの違いが競争力の源泉（一朝一夕には構築できないもの）
- DX取り組み領域の見極め
  - 事業
    1. 顧客・価値
    2. ビジネスモデル
    3. オペレーション
    4. 事業インフラ（立地／建屋・ITシステム・マテハン：荷役を効率化するシステムの総称）
    5. 人材・組織
    - オペレーション・事業インフラがDXの取り組み対象
    - 活動システムマップを活用

### 3-4 DXへの取り組みポイントの見極め

- 活動システムマップ
    >活動システムマップ（Activity System Map）は、マイケル・ポーターが提唱した、企業の競争優位性を生み出す「一連の活動（業務や戦略）」とその「相互関係」を可視化するフレームワークです。
- 活動システムマップでDXの取り組み対象を絞り込む

### 3-5 本書における物流DXの定義

- 本書における物流DXの定義
  1. DX：競争優位を実現する経営・業務における変革を、データとデジタル技術という手段を使って実現すること
  2. プロセスイノベーション：デジタル技術を活用して、自社・業界の業務プロセスを変革すること。業務効率を高めるBPRを実現し、主に事業の運営コストを削減、利益の改善に寄与すること。
  3. 物流：サプライチェーンにおいて物流がボトルネックであり、この解消が顧客価値を生み事業の競争優位をつくることから、事業戦略の重要ファクターになっている
  >「事業の競争優位をつくる物流の在り方（物流戦略）をオペレーション起点で構想し、それをデータとデジタル技術を用いて実現すること」
- 持続性ある物流DXが必要：常に変化に対応し続ける

### 3-6 経営者の積極的な参画が推進力を生む

- 社内のコンフリクトを解消するのは経営者しかいない
  - 経営者による情報発信が重要
  - 経営者による積極的なコミットメントが必要不可欠

## 4章 物流DXの進め方

### 4-1 物流DXの活動ステップ

- 物流DXの活動ステップ（1～4の繰り返し）
  1. 構想策定
     - 目的・目標の設定
     - 課題抽出と対策方向性
     - 新オペレーションデザイン
  2. 詳細検討
     - オペレーション設計
     - 打ち手への要件決定
     - 打ち手ベンダー選定
  3. 打ち手構築・導入
     - 打ち手別の要件定義～構築
     - 新オペレーションへの準備
     - 導入・移行
  4. 新オペレーション稼働
     - オペレーション安定化
     - 次段階への課題抽出

### 4-2 構想策定の概要

- 構想書
  1. 目的・目標
     - 何のため、何を実現するのか
  2. 目標実現に向けた課題
     - 定性調査
     - 定量調査
  3. 課題解決オペレーション
     - 全体最適
  4. 打ち手案
  5. 投資対効果
  6. 取り組みロードマップ

### 4-3 構想により骨組みを作る

- 具体性をもった構想が重要
  - 課題A：ピッキング作業を効率化する
  - 課題B：ピッキング作業時間の60%を占める歩行について削減策を講じる ← より具体的

### 4-4 近すぎず遠すぎない目標設定

- 実現可能性がある高い目標
  - SMARTの法則
    1. Specific：具体的である
    2. Measurable：計測が可能である
    3. Achievable：達成可能な水準である
    4. Relevant：上位目標につながる下位目標になっている
    5. Time-bound：明確な期限を定めている

### 4-5 現状実態を明らかにする

- 実体はわかっているようでいてわかっていない
  - モノ・作業・情報の流れを知る
    1. 業務フロー：業務の流れとつながりを記述したもの
    2. 物量フロー：モノの流れと、その流量を記述したもの
    3. システム構想図：基幹システム・WMS・TMSなど、各システムの機能とデータのつながりを記述したもの
  - インタビューは、その場でまとめる

### 4-6 現状分析と課題整理

- 現状分析
  - 定性分析：数値化できない要素の分析（インタビュー等）
    - 現象：現状調査により集まった問題点
    - 真因：隠された本質的問題点
  - 定量分析：数値化できる要素の分析
- 課題
  - トップダウン課題
  - ボトムアップ課題

### 4-7 将来を支える新オペレーション

- 整流化：極力シンプルな業務プロセス（きれいな流れ）
  - 計画・ 実行・管理が大事
  - 作業方式を選定した上でその手段を選ぶ

### 4-8 詳細検討の概要

- 新オペレーションの詳細化
  - モノ・作業・情報の流れを詳細化
- 新オペレーションを実現する打ち手の要件を洗い出す
- 各打ち手の要件ボリュームを基にロードマップを描く
- コンペティションによるベンダー選定
  - 提案依頼書（RFP：Request for Proposal）

### 4-9 止めない物流のためのオペレーション設計

- 障害に強いオペレーションをつくる
  - 止めない物流に向けたオペレーション設計
    - 自然災害：地震・洪水 ← 倉庫分散
    - 設備の障害 ← 複数の障害レベルに応じた対策案を検討
  - 障害時の「逃げ」ができる設計を実施する
    - 端末トラブル：他の端末でも作業可能
    - 紙ベースでの作業可能
    - サーバー、ネットワークの冗長化
    - マニュアルでも出荷が可能

### 4-10 構築の主体はユーザー企業

- ×：ベンダー任せ
- 〇：ユーザー企業の主体的な行動が不可欠
- 物流DX部門の基盤
  - DX構想～実行メンバーは、貴重な人材：継続的に選任するべき

### 4-11 導入準備は先行して実施する

- 専属メンバーでない現場の協力が不可欠
- 成功を左右する要件定義と受入テスト
- V字モデル

  ```text
  要件定義　-------------->　受入テスト
  　　基本設計　　　　　　総合、結合テスト
  　　　　詳細設計　　単体テスト
  　　　　　　　　開発
  ```

## 5章 データドリブンPDCAを埋め込む

### 5-1 データドリブン＝データに基づく意思決定

- KKD（勘・経験・度胸）による意思決定の限界
  - 小さいビジネスならKKDも大きな失敗はないかも
  - 大きなビジネスでは不可
- データドリブン経営
  - データに基づく意思決定
    1. KKDに根拠を与えられる
    2. 個人技から脱却できる
    3. 広く・深い検討ができる

### 5-2 データドリブン経営の目的と戦略

- 目的（戦略）ありき
  - データドリブン：手段
  - 目的がないと判断できない
  - 目的の実行状況をデータで判断する

### 5-3 必要なデータを絞り込む

- 必要なデータを見極める
  - データサイエンティストが何でもできるわけではない
  - 必要なデータを知っているのは自分自身

### 5-4 意思決定の階層毎に必要なデータは異なる

- あらかじめ各階層で必要なデータを定義しておく
- 意思決定の階層と必要なデータ
  1. 上級管理層（CLO：Chief Logistics Officerや物流本部長 など）
     - 物流構造やそのQCD・人事に至る広範囲なデータ
  2. 中間管理層（物流部長や物流センター長 など）
     - 業務実績（QCD）データが中心
  3. オペレーション層（物流担当や現場リーダー など）
     - 作業データと担当範囲における業務実績（QCD）データ

### 5-5 データドリブン経営の実践ステップ

- データドリブンの実践（1～5の繰り返し）

     - データ活用範囲の設定
     - 達成目標の設定
  2. データ収集
     - データの構造化・体系化
     - データの一元化
     - データ収集のルール化
  3. データ分析
     - データ整理
     - 分析視点の設定
  4. 結果の可視化
     - 対象者の明確化
     - 論点に沿ったビジュアル化
  5. 評価・意思決定
     - 目的・目標の明確化
     - 判断基準と判断方法の設定
     - 管理・統制サイクルの設定

### 5-6 物流におけるデータドリブンの重要性

- 物流現場では、データ活用が進んでいない
  1. 日々の業務に忙殺されている
  2. データ収集が容易でない
  3. データ活用手法が浸透していない
